# BNV Manager v2

## Project Overview
Club management platform (Vereinsverwaltung) for managing mail accounts via Mailcow. Built for elderly users — UI must be simple and clear. Three roles: SuperAdmin, ClubAdmin, ClubMember.

## Tech Stack
- **Backend**: Rust, Galvyn framework, RORM ORM, PostgreSQL
- **Frontend**: React 19, TypeScript, Vite, TanStack Router (file-based), TanStack Form, TailwindCSS 4, HeadlessUI
- **Mail**: Mailcow integration via custom SDK crate (`mailcow/`)
- **Auth**: OIDC (Keycloak). Mailboxes are created via OIDC provisioning, NOT by this app.
- **Deployment**: Docker Compose, GHCR images, nginx reverse proxy

## Architecture Notes

### Roles & Auth
- **SuperAdmin** (`AdministrativeAccount`): Full system access, no club affiliation
- **ClubAdmin** (`ClubAdminAccount`): Manages one club, scoped by club_uuid middleware
- **ClubMember** (`ClubAccount`): Regular member, belongs to one club
- Session stores only AccountUuid; role determined dynamically per request

### Mailcow Integration
- Mailboxes are created via OIDC, NOT by the app
- App manages: app passwords, domain admins, mailbox deletion, reading stats
- `SyncWorker` periodically syncs domain list from Mailcow
- `AppPasswordInitializer` sets app passwords after password changes (sleeps 5s waiting for OIDC mailbox creation)
- Some Mailcow API fields are strings (bytes_total, msgs_total) — need custom deserialization

### API Structure
- `/api/v1/frontend/admin/...` — SuperAdmin endpoints (auth_superadmin middleware)
- `/api/v1/frontend/club-admin/clubs/{club_uuid}/...` — ClubAdmin endpoints (auth_club_admin middleware)
- `/api/v1/frontend/common/...` — Authenticated common endpoints
- OpenAPI specs served at `/api/v1/frontend/{admin,club-admin,common,auth}.json` (debug only)

### Frontend Patterns
- Generated API clients in `src/api/generated/` — regenerated by `npm run gen-api`, manual edits get overwritten
- API wrappers in `src/api/api_admin.tsx`, `src/api/api_club_admins.tsx` — these are the stable interface
- Translations in `public/locales/{de,en}/` — namespace per page/dialog
- All components need JSDoc (eslint enforces 0 warnings)
- `Field` component uses `data-slot` CSS selectors for spacing — wrapping inputs needs `data-slot="control"` on the wrapper

### Code Style
- Backend: `cargo +nightly fmt` (nightly rustfmt)
- Frontend: Prettier + ESLint with JSDoc plugin (0 warnings policy)
- All exported types/functions need JSDoc with `@param`/`@returns` where applicable
- Inline prop types on components trigger JSDoc warnings — extract to named types
- German is the primary locale, English secondary
